# FinVault Hackathon — Full Project Blueprint

**FinVault** is a personal finance and banking web application that runs fully locally on Replit. It *looks* complete and polished, but contains three deeply embedded bugs that require genuine investigation and two partially stubbed features to implement. The entire project is structured around GitHub Issues released on a timer throughout a 3–4 hour session.

***

## Tech Stack

| Layer | Technology | Why |
|---|---|---|
| Frontend | HTML + Tailwind CSS (CDN) + Alpine.js (CDN) + Chart.js (CDN) | Zero build step on Replit, elegant UI, reactive without React overhead |
| Backend | FastAPI + Uvicorn (Python) | Modern async Python, auto-generated docs at `/docs`, easy for Copilot |
| ORM | SQLAlchemy 2.0 (async) | Industry standard, supports async properly |
| Database | SQLite (local) → PostgreSQL-ready | Zero config for Replit, Alembic handles migration portability |
| Migrations | Alembic | Version-controlled schema history stored in repo |
| Auth | python-jose (JWT) + passlib (bcrypt) | Standard JWT access/refresh token pattern |
| Testing | pytest + pytest-asyncio + httpx | Async-capable test suite with built-in timing assertions |

## 🐛 Bug Challenge 1 — Race Condition on Fund Transfer

**GitHub Issue Title**: `[BUG] Concurrent transfers causing account overdrafts — balance going negative`

**Released**: T+0 (at hackathon start)

**What participants observe**: Running the provided test produces accounts with negative balances. Manually triggering two rapid transfers from the same account can overdraft it. The `GET /accounts/{id}` endpoint shows a balance of `-€120` on an account that started at `€100`.

**Where the bug lives** — `backend/app/services/transfer_service.py`:

```python
# ❌ BUGGY: Classic check-then-act race condition
async def transfer_funds(from_id: int, to_id: int, amount: float, db: AsyncSession):
    from_acct = await db.get(Account, from_id)       # READ
    to_acct   = await db.get(Account, to_id)

    if from_acct.balance < amount:
        raise HTTPException(status_code=400, detail="Insufficient funds")

    # ⚠️ Another concurrent request reads the same balance here
    await asyncio.sleep(0.008)  # Simulates real processing latency

    from_acct.balance -= amount   # WRITE (too late — balance already stale)
    to_acct.balance   += amount
    await db.commit()
```

The `asyncio.sleep` is intentionally placed mid-operation to widen the race window and make it reliably reproducible without needing extreme concurrency. [kingtigerebooks.co](https://kingtigerebooks.co.uk/post/race-condition-using-fast-api-sql-alchemy-and-asyncpg)

**The test** — `tests/test_transfer_race.py`:

```python
import asyncio, pytest
import httpx
from app.main import app

@pytest.mark.asyncio
async def test_concurrent_transfers_no_overdraft():
    """Account starts at €100. Ten concurrent €20 transfers are fired.
    Only 5 should succeed. Final balance must be ≥ 0."""
    async with httpx.AsyncClient(app=app, base_url="http://test") as client:
        token = await login(client, "alice@finvault.io")
        tasks = [
            client.post("/api/transfer",
                json={"from_account_id": 1, "to_account_id": 2, "amount": 20.0},
                headers={"Authorization": f"Bearer {token}"}
            ) for _ in range(10)
        ]
        responses = await asyncio.gather(*tasks)

    final_balance = await get_balance(1)
    assert final_balance >= 0, \
        f"Race condition: balance is {final_balance} — overdraft occurred"
    
    successes = sum(1 for r in responses if r.status_code == 200)
    assert successes == 5, f"Expected 5 successful transfers, got {successes}"
```

**The fix** — participants must implement row-level locking using SQLAlchemy's `with_for_update()` inside an atomic transaction, preventing any other session from reading the same row between the SELECT and the UPDATE:

```python
# ✅ FIXED: Atomic SELECT FOR UPDATE within a transaction
async with db.begin():
    from_acct = (await db.execute(
        select(Account).where(Account.id == from_id).with_for_update()
    )).scalar_one()
    # ... rest of logic
```

**Scoring signal**: Test suite goes from ❌ → ✅. AI-assisted teams should use Copilot to understand `SELECT FOR UPDATE` and SQLAlchemy async transactions. [kingtigerebooks.co](https://kingtigerebooks.co.uk/post/race-condition-using-fast-api-sql-alchemy-and-asyncpg)

## 🐛 Bug Challenge 2 — N+1 Query Problem on Transaction History

**GitHub Issue Title**: `[BUG] /transactions/{account_id} endpoint degrading to 4–8 seconds response time`

**Released**: T+60 minutes

**What participants observe**: Loading the transaction history page feels fast on first load (few records), but the attached timing test fails consistently. When they check the logs, they see hundreds of individual SQL queries firing per request. The seed data includes 500+ transactions per account specifically to make this obvious. [oneuptime](https://oneuptime.com/blog/post/2026-02-02-fastapi-async-database/view)

**Where the bug lives** — `backend/app/routers/transactions.py`:

```python
# ❌ BUGGY: N+1 queries — 1 query for transactions + N for categories + N for accounts
@router.get("/transactions/{account_id}")
async def get_transactions(account_id: int, db: AsyncSession = Depends(get_db)):
    result = await db.execute(
        select(Transaction).where(Transaction.account_id == account_id)
        # Missing: .options(selectinload(Transaction.category))
        # Missing: no pagination
        # Missing: no index on account_id column
    )
    transactions = result.scalars().all()

    enriched = []
    for txn in transactions:
        category = await db.get(Category, txn.category_id)  # 🔥 N queries
        enriched.append({
            "id": txn.id, "amount": txn.amount,
            "category": category.name,             # Fired separately per row
            "date": txn.date.isoformat()
        })
    return enriched
```

There is also a missing index baked into the initial migration intentionally: the `transactions` table's `account_id` column has **no index**, meaning each query does a full table scan. [nashruddinamin](https://www.nashruddinamin.com/blog/how-to-use-alembic-for-database-migrations-in-your-fastapi-application)

**The test** — `tests/test_transaction_performance.py`:

```python
import time, pytest
from sqlalchemy import event

query_count = 0

@pytest.fixture(autouse=True)
def count_queries(engine):
    global query_count
    query_count = 0
    @event.listens_for(engine.sync_engine, "before_cursor_execute")
    def _count(conn, cursor, statement, *args):
        global query_count
        query_count += 1
    yield
    event.remove(engine.sync_engine, "before_cursor_execute", _count)

def test_transaction_list_response_time():
    start = time.perf_counter()
    response = client.get("/api/transactions/1", headers=auth_header)
    elapsed_ms = (time.perf_counter() - start) * 1000
    assert response.status_code == 200
    assert elapsed_ms < 300, f"Too slow: {elapsed_ms:.0f}ms (N+1 queries suspected)"

def test_transaction_query_count():
    client.get("/api/transactions/1", headers=auth_header)
    assert query_count <= 3, \
        f"N+1 detected: {query_count} queries fired (should be ≤ 3)"
```

**The fix requires two steps**, which is what makes this non-trivial:
1. Replace the loop with `selectinload` or `joinedload` on the relationship
2. Write and run a new Alembic migration adding an index: `op.create_index('ix_transactions_account_id', 'transactions', ['account_id'])`

**Scoring signal**: Both assertions in the test pass. Teams learn SQLAlchemy eager loading and understand the index/query relationship deeply. [youtube](https://www.youtube.com/watch?v=N9y9QkBM-Aw)

## 🐛 Bug Challenge 3 — Refresh Token Reuse Vulnerability + In-Memory Blacklist

**GitHub Issue Title**: `[BUG] Security: Refresh tokens can be reused by concurrent requests + logout doesn't survive server restart`

**Released**: T+90 minutes

**What participants observe**: Two security test cases both fail. First, concurrent requests using the same refresh token all succeed — meaning one stolen token grants unlimited new access tokens. Second, logging out a user and then restarting the Replit server (clearing memory) makes the old token valid again. [stackoverflow](https://stackoverflow.com/questions/78718367/race-condition-using-fastapi-sqlalchemy-and-asyncpg)

**Where the bug lives** — `backend/app/middleware/auth_middleware.py` and `backend/app/routers/auth.py`:

```python
# ❌ BUGGY: In-memory blacklist — wiped on every server restart
_token_blacklist: list[str] = []   # Should be a DB table, not a module variable

def blacklist_token(token: str):
    _token_blacklist.append(token) # Not thread-safe under high concurrency

def is_blacklisted(token: str) -> bool:
    return token in _token_blacklist  # O(n) scan

# ❌ BUGGY: TOCTOU race on refresh token rotation
@router.post("/auth/refresh")
async def refresh(body: RefreshRequest, db: AsyncSession = Depends(get_db)):
    record = await db.execute(
        select(RefreshToken).where(RefreshToken.token == body.refresh_token)
    )
    token_obj = record.scalar_one_or_none()

    if not token_obj or token_obj.is_revoked:
        raise HTTPException(401, "Invalid token")

    # ⚠️ Window here: concurrent request reads same un-revoked token
    await asyncio.sleep(0.005)

    token_obj.is_revoked = True   # Too late — race already won
    await db.commit()
    return {"access_token": create_access_token(token_obj.user_id)}
```

**The test** — `tests/test_auth_security.py`:

```python
@pytest.mark.asyncio
async def test_refresh_token_single_use_only():
    """A refresh token must only produce ONE new access token, even under concurrency."""
    refresh_tok = await get_refresh_token("alice@finvault.io")
    async with httpx.AsyncClient(app=app, base_url="http://test") as client:
        tasks = [
            client.post("/api/auth/refresh", json={"refresh_token": refresh_tok})
            for _ in range(6)
        ]
        responses = await asyncio.gather(*tasks)
    
    successes = [r for r in responses if r.status_code == 200]
    assert len(successes) == 1, \
        f"Token reuse: {len(successes)} new tokens issued from 1 refresh token"

def test_logout_survives_server_restart(client):
    token = login_and_get_token("bob@finvault.io")
    client.post("/api/auth/logout", headers={"Authorization": f"Bearer {token}"})

    # Simulate restart by clearing the in-memory list
    import app.middleware.auth_middleware as mw
    mw._token_blacklist.clear()   # This is exactly what a Replit restart does

    response = client.get("/api/accounts", headers={"Authorization": f"Bearer {token}"})
    assert response.status_code == 401, "Revoked token accepted after simulated restart"
```

**The fix requires two changes**:
1. Create and run a new Alembic migration adding a `revoked_tokens` table (`token TEXT, revoked_at TIMESTAMP`), then update blacklist functions to read/write from DB
2. Wrap the refresh token rotation in `SELECT FOR UPDATE` inside a transaction, same pattern as Bug #1

This challenge deliberately echoes the patterns from Bugs #1 and #2 to reinforce learning. [stackoverflow](https://stackoverflow.com/questions/78718367/race-condition-using-fastapi-sqlalchemy-and-asyncpg)

## Key Design Principles Behind the Bugs

All three bugs share intentional design choices that make them non-trivial:

- **None are syntax errors or missing imports** — the app runs fine and looks correct on casual inspection
- **Each bug requires understanding async concurrency, database internals, or security patterns** — exactly the depth that makes Copilot/AI useful as an analysis tool rather than a trivial autocomplete
- **Each bug has a failing test already written** — participants don't need to write tests, they need to understand why the test fails and fix root cause
- **Bugs #1 and #3 share the same fix pattern** (`SELECT FOR UPDATE`) applied in two different contexts — reinforcing the pattern across auth and financial domains
- **The `asyncio.sleep()` lines are commented with `# Simulates processing delay`** — honest enough not to mislead but not flagged as a hint [reddit](https://www.reddit.com/r/FastAPI/comments/1m14r2k/when_to_worry_about_race_conditions/)

from datetime import datetime, timedelta, timezone
from jose import JWTError, jwt
from passlib.context import CryptContext
from app.config import settings
import secrets

pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

def hash_password(password: str) -> str:
    return pwd_context.hash(password)

def verify_password(plain: str, hashed: str) -> bool:
    return pwd_context.verify(plain, hashed)

def create_access_token(user_id: int) -> str:
    expire = datetime.now(timezone.utc) + timedelta(minutes=settings.ACCESS_TOKEN_EXPIRE_MINUTES)
    return jwt.encode(
        {"sub": str(user_id), "exp": expire, "type": "access"},
        settings.SECRET_KEY, algorithm=settings.ALGORITHM
    )

def create_refresh_token() -> str:
    return secrets.token_urlsafe(64)

def decode_access_token(token: str) -> dict:
    return jwt.decode(token, settings.SECRET_KEY, algorithms=[settings.ALGORITHM])

async def get_current_user(
    credentials: HTTPAuthorizationCredentials = Depends(security),
    db: AsyncSession = Depends(get_db),
) -> User:
    token = credentials.credentials
    if is_blacklisted(token):
        raise HTTPException(status_code=401, detail="Token has been revoked")
    try:
        payload = decode_access_token(token)
        user_id = int(payload["sub"])
    except (JWTError, ValueError, KeyError):

    result = await db.execute(select(User).where(User.id == user_id, User.is_active == True))
    user = result.scalar_one_or_none()
    if not user:
        raise HTTPException(status_code=401, detail="User not found")
    return user

# ==========================================================================
# BUG #3 — IN-MEMORY TOKEN BLACKLIST
# ==========================================================================
# PROBLEM 1 (Persistence): This list lives in Python process memory.
#   Every Replit restart, every uvicorn --reload, every crash wipes it.
#   A user who logs out gets their token revoked in memory only.
#   After any restart that same token is valid again.
#
# PROBLEM 2 (Thread safety): list.append() is not safe under asyncio
#   concurrent requests — the list can be read mid-write.
#
# PROBLEM 3 (Performance): `token in list` is O(n) linear scan.
#   With thousands of revoked tokens this endpoint slows to a crawl.
#
# HOW PARTICIPANTS FIX IT:
#   Step 1 — Write a new Alembic migration adding table:
#     revoked_tokens(id PK, token TEXT UNIQUE, revoked_at TIMESTAMP)
#   Step 2 — Replace blacklist_token() to INSERT into that table
#   Step 3 — Replace is_blacklisted() to SELECT COUNT(*) from that table
#   Step 4 — Run: alembic upgrade head
# ==========================================================================

_token_blacklist: list[str] = []  # ← cleared on every server restart


def blacklist_token(token: str) -> None:
    """Revoke an access token. WARNING: not persistent across restarts."""
    _token_blacklist.append(token)  # not thread-safe under async concurrency


def is_blacklisted(token: str) -> bool:
    """Check if a token is revoked. WARNING: returns False after any restart."""
    return token in _token_blacklist  # O(n) scan

    #             .where(RefreshToken.token == body.refresh_token)
    #             .with_for_update()    ← acquires row-level lock

THIS FILE CONTAINS BUG #1 — race condition on concurrent transfers

import asyncio
from sqlalchemy.ext.asyncio import AsyncSession
from sqlalchemy import select
from fastapi import HTTPException
from app.models.account import Account
from app.models.transaction import Transaction
from app.models.category import Category


async def transfer_funds(
    from_account_id: int,
    to_account_id: int,
    amount: float,
    description: str,
    db: AsyncSession,
) -> dict:
    """
    Transfer funds between two accounts.

    ==========================================================================
    BUG #1 — CHECK-THEN-ACT RACE CONDITION (no row-level locking)
    ==========================================================================
    WHAT HAPPENS under concurrent requests:
      Request A: reads from_account.balance = €500.00
      Request B: reads from_account.balance = €500.00  ← same stale value
      Request A: balance (500) >= amount (500) → passes check
      Request B: balance (500) >= amount (500) → passes check ← BOTH PASS
      asyncio.sleep() widens the window — control yields here
      Request A: writes balance = 500 - 500 = €0.00
      Request B: writes balance = 500 - 500 = €0.00  ← OVERDRAFT: -€500!
      Both commits succeed. Account balance is now -€500.

    VISIBLE SYMPTOM: Running the test fires 10 concurrent €50 transfers
    from an account with €500. Only 10 should succeed with balance = €0,
    but you'll see accounts go negative. The test fails with:
      AssertionError: Race condition: final balance is -€450.00

    HOW PARTICIPANTS FIX IT:
      Acquire a row-level lock using SELECT FOR UPDATE before any read:

      async with db.begin():
          from_account = (await db.execute(
              select(Account)
              .where(Account.id == from_account_id)
              .with_for_update()   ← blocks concurrent reads of this row
          )).scalar_one_or_none()

          to_account = (await db.execute(
              select(Account)
              .where(Account.id == to_account_id)
              .with_for_update()
          )).scalar_one_or_none()

          # now check and update safely — no other session can read these rows
          if from_account.balance < amount:
              raise HTTPException(400, "Insufficient funds")
          from_account.balance -= amount
          to_account.balance   += amount
          db.add(transaction_record)
          # db.begin() context manager commits here automatically
    ==========================================================================
    """

    if amount <= 0:
        raise HTTPException(400, "Transfer amount must be positive")
    if from_account_id == to_account_id:
        raise HTTPException(400, "Cannot transfer to the same account")

    # READ without any lock — race condition window is open
    from_account = (await db.execute(
        select(Account).where(Account.id == from_account_id)
    )).scalar_one_or_none()

    to_account = (await db.execute(
        select(Account).where(Account.id == to_account_id)
    )).scalar_one_or_none()

    if not from_account:
        raise HTTPException(404, "Source account not found")
    if not to_account:
        raise HTTPException(404, "Destination account not found")

    if from_account.balance < amount:
        raise HTTPException(400, f"Insufficient funds. Available: €{from_account.balance:.2f}")

    # Simulates real-world processing latency (e.g. fraud check API call).
    # This widens the race window significantly and makes the bug reliably reproducible.
    await asyncio.sleep(0.010)

    # WRITE — the balance read above is now stale; another request may have modified it
    from_account.balance = round(from_account.balance - amount, 2)
    to_account.balance = round(to_account.balance + amount, 2)

    transfer_category = (await db.execute(
        select(Category).where(Category.name == "Transfers")
    )).scalar_one_or_none()

    import secrets as _secrets
    ref = _secrets.token_hex(6).upper()

    db.add(Transaction(
        account_id=from_account_id,
        category_id=transfer_category.id if transfer_category else 1,
        amount=amount,
        transaction_type="debit",
        description=description or f"Transfer to ****{str(to_account.account_number)[-4:]}",
        reference=f"TRF-{ref}",
    ))
    db.add(Transaction(
        account_id=to_account_id,
        category_id=transfer_category.id if transfer_category else 1,
        amount=amount,
        transaction_type="credit",
        description=description or f"Transfer from ****{str(from_account.account_number)[-4:]}",
        reference=f"TRF-{ref}",
    ))
    await db.commit()

    return {
        "message": "Transfer successful",
        "reference": f"TRF-{ref}",
        "from_balance": from_account.balance,
        "to_balance": to_account.balance,
    }

@router.get("")
async def list_budgets(current_user: User = Depends(get_current_user), db: AsyncSession = Depends(get_db)):
    today = date.today()
    start = today.replace(day=1)

    budgets_result = await db.execute(
        select(Budget)
        .options(selectinload(Budget.category))
        .where(Budget.user_id == current_user.id)
    )
    budgets = budgets_result.scalars().all()

    account_ids = [r[0] for r in (await db.execute(
        select(Account.id).where(Account.user_id == current_user.id)
    )).fetchall()]

    result = []
    for b in budgets:
        spent_result = await db.execute(
            select(func.coalesce(func.sum(Transaction.amount), 0.0))
            .where(
                Transaction.account_id.in_(account_ids),
                Transaction.category_id == b.category_id,
                Transaction.transaction_type == "debit",
                Transaction.date >= start,
            )
        )
        spent = float(spent_result.scalar())
        result.append({
            "id": b.id, "period": b.period,
            "limit_amount": b.limit_amount,
            "spent": round(spent, 2),
            "remaining": round(max(b.limit_amount - spent, 0), 2),
            "percentage_used": round(min((spent / b.limit_amount) * 100, 100), 1) if b.limit_amount > 0 else 0,
            "category": {"id": b.category.id, "name": b.category.name, "icon": b.category.icon, "colour_hex": b.category.colour_hex},
        })
    return {"budgets": result}

    url = config.get_main_option("sqlalchemy.url")
    context.configure(url=url, target_metadata=target_metadata, literal_binds=True, dialect_opts={"paramstyle": "named"})
    with context.begin_transaction():
        context.run_migrations()


def do_run_migrations(connection):
    context.configure(connection=connection, target_metadata=target_metadata)
    with context.begin_transaction():
        context.run_migrations()

    connectable = async_engine_from_config(
        config.get_section(config.config_ini_section, {}),
        prefix="sqlalchemy.",
        poolclass=pool.NullPool,
    )
    async with connectable.connect() as connection:
        await connection.run_sync(do_run_migrations)
    await connectable.dispose()

    run_migrations_offline()
else:
    asyncio.run(run_migrations_online())

const API_BASE = "/api";

function getToken() { return localStorage.getItem("finvault_token"); }
function setTokens(access, refresh) {
  localStorage.setItem("finvault_token", access);
  localStorage.setItem("finvault_refresh", refresh);
}
function clearTokens() {
  localStorage.removeItem("finvault_token");
  localStorage.removeItem("finvault_refresh");
}

async function apiFetch(path, options = {}) {
  const headers = { "Content-Type": "application/json", ...options.headers };
  const token = getToken();
  if (token) headers["Authorization"] = `Bearer ${token}`;

  const response = await fetch(`${API_BASE}${path}`, { ...options, headers });

  if (response.status === 401) {
    // Attempt token refresh, then retry once
    const refreshed = await tryRefreshToken();
    if (refreshed) {
      headers["Authorization"] = `Bearer ${getToken()}`;
      return fetch(`${API_BASE}${path}`, { ...options, headers });
    } else {
      clearTokens();
      window.dispatchEvent(new CustomEvent("finvault:logout"));
      throw new Error("Session expired");
    }
  }

  return response;
}

async function tryRefreshToken() {
  const rt = localStorage.getItem("finvault_refresh");
  if (!rt) return false;
  try {
    const r = await fetch(`${API_BASE}/auth/refresh`, {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ refresh_token: rt }),
    });
    if (!r.ok) return false;
    const data = await r.json();
    setTokens(data.access_token, data.refresh_token);
    return true;
  } catch {
    return false;
  }
}

// Expose global helpers
window.FinVaultAPI = {
  get:    (path)         => apiFetch(path),
  post:   (path, body)   => apiFetch(path, { method: "POST", body: JSON.stringify(body) }),
  put:    (path, body)   => apiFetch(path, { method: "PUT", body: JSON.stringify(body) }),
  delete: (path)         => apiFetch(path, { method: "DELETE" }),
  setTokens, clearTokens, getToken,
};
